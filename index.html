LOGO! 0BA5 / 0BA6 Program Dumper

Reads the circuit-program memory off a Siemens LOGO! 0BA5 or 0BA6 over the serial PG protocol, in your browser (Chrome/Edge, desktop or Android). Two transports: Web Serial (the OS driver configures the cable — most reliable) and WebUSB (this tool configures the chip directly — needed where Web Serial doesn't list the cable). Nothing is uploaded — it all runs locally.

Works with: LOGO! 0BA6 (0BA6 / 0BA6.ES3 / 0BA6.ES10, part 6ED1052-…-0BA6) and 0BA5 — the generations with the 4-wire RS-232 PG serial interface (9600 8E1). The PLC is identified after the cable connects. 0BA5 is implemented but not yet verified on real hardware; its legacy image can be decoded, while 0BA6 is currently saved raw.
Does NOT work with: LOGO! 0BA7 / 0BA8 (LOGO!8) — those use Ethernet, a completely different protocol; and 0BA4 / earlier.
Cable / serial: the USB LOGO! programming cable (or any USB-serial adapter — FTDI, CP210x, CH340/CH341, Prolific). Web Serial (desktop Chrome/Edge) works with any chip the OS has a driver for; WebUSB drives the chip itself — CH340/CH341 is verified, CP210x and FTDI are best-effort/unverified. Browsers: Chrome/Edge on desktop, or Chrome on Android via a USB-OTG adapter. Safari, Firefox, and iOS are not supported (no Web Serial / WebUSB).

Before you start — plug the USB cable in, and put the LOGO! in STOP (ESC → Stop on the device; step 2 also does it). On Android use an OTG adapter and Chrome. Follow the numbered steps — the glowing button is the recommended next one. The log records commands, replies, and long-read progress; use Copy log to share it.

Do these in order. Steps 4 and 6 write to the PLC and require a second click to confirm.

not connected

Steps 3–4 are only needed if the program is password-protected. Step 3 reads and displays the stored password (cleartext on older models, XOR-decoded on ES10) — it never writes to the PLC, so read it and verify it looks right. Step 4 then clears protection so step 5 can read; if the store is hidden, that write is explicitly labelled an experimental bypass. Step 6 restores protection. No password → skip to step 5.

      Debugging aids. Mode picks the read regime — the two are NOT interchangeable, and the wrong one reads zeros:
      
Memory = program/data memory: Read Block (0x05) on the 0BA6, byte reads on the 0BA5, with a bare address (never paged). Use it for the program image — e.g. the complete 0BA6 upload is 13464 bytes from the bare 0x00002FAA (body 0x00003292); the 0BA5 image is 0x0C14/0x0E20/0x0EE8.
      
Register = symbolic register: Read Byte (0x02) with the getAdress page rule applied, so a base 48FF reads the paged 0x00FF48FF on a 0BA6 (a 0BA5 drops the page automatically). Use it for the flag/magic/ident/firmware/protection registers, the password store 0566, or the program name 0570.
      
The detected model fills in sensible defaults per mode; edits you type are kept. Uniform 0x00/0xFF captures are saved because they can prove protection, emptiness, or a wrong map.
